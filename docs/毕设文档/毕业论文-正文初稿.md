# 基于 LBS 的微信小程序校园二手交易系统设计与实现

## 摘要

随着移动互联网的发展和校园闲置物品流转需求的增加，二手交易已经成为高校中比较常见的生活场景。学生在毕业、搬宿舍、更新电子设备时会产生不少闲置物品，但传统交易方式大多依赖群聊、熟人转让或线下询问，信息分散、匹配效率不高，交易过程也不够方便。通用型二手平台虽然功能较全，但在校园近距离交易、本地化筛选和信任关系方面并不完全贴合实际使用习惯。基于这一背景，本文设计并实现了一套基于 LBS 的微信小程序校园二手交易系统。

系统采用前后端分离方案。前端使用 Taro、React 和 TypeScript 开发微信小程序，后端基于 Flask、SQLAlchemy 和 MySQL 实现业务服务，通过 JWT 完成身份认证，并结合 WebSocket 作为即时通信的扩展方案。围绕校园二手交易这一场景，系统实现了用户管理、商品发布、附近检索、聊天会话、订单管理、交易评价、信用分、举报和浏览历史等模块。其中，针对“就近交易”这一比较实际的需求，后端实现了基于 Haversine 公式的距离计算与附近排序逻辑，支持结合用户当前位置进行范围筛选、距离排序和地图展示。另外，结合项目后续扩展需要，论文中也补充了后台管理系统的设计方案，用于支撑商品审核、举报处理、用户封禁、分类维护和运营统计等功能。

从当前实现情况来看，系统已经完成了用户端的主要业务流程，基本可以覆盖校园用户发布商品、查找附近商品、沟通交易和完成评价等常见操作。引入 LBS 能力之后，商品检索方式不再只是按时间排序，而是更贴近校园线下面交的实际需求。对于毕业设计来说，这样的实现已经能较完整地体现一个校园交易平台的核心思路。

关键词：LBS；微信小程序；校园二手交易；Taro；Flask

## Abstract

With the rapid development of mobile Internet and the sharing economy, second-hand trading has become an important way to recycle idle resources on campus. In universities, students often generate a large number of unused goods when graduating, changing dormitories, or upgrading electronic devices. Traditional offline transactions suffer from limited information dissemination, low matching efficiency, and inconvenient trading processes. Although general second-hand platforms provide basic product display and communication functions, they are still insufficient in localized matching, nearby transactions, and trust building for campus scenarios. Therefore, this paper designs and implements a campus second-hand trading system based on LBS and WeChat Mini Program.

The system adopts a front-end and back-end separated architecture. The front end is developed with Taro, React, and TypeScript to build a WeChat Mini Program, while the back end is implemented with Flask, SQLAlchemy, and MySQL. JWT is used for user authentication, and WebSocket is introduced for real-time messaging. In terms of business design, the system includes user management, goods publishing, nearby search, instant messaging, order management, transaction evaluation, credit score, reporting, and browsing history. To meet the practical requirement of nearby matching in campus transactions, the system implements distance calculation and sorting based on the Haversine formula, enabling range filtering, distance-based ranking, and map visualization according to the user's current location. In order to improve system completeness and governance capability, this paper also proposes an extended design for an administration system, which supports product review, report handling, user banning, category management, and operational statistics.

The implementation results show that the system completes the core business loop on the mobile side and can satisfy users' needs in publishing goods, finding nearby products, communicating with sellers, and finishing transactions. The study indicates that introducing LBS technology into the campus second-hand trading scenario can effectively improve retrieval efficiency and local transaction convenience. This work has practical value for the design and implementation of campus idle resource circulation platforms and provides a reference for similar lightweight trading systems based on location services.

Keywords: LBS; WeChat Mini Program; Second-hand Trading; Campus Platform; Flask

# 第一章 绪论

## 1.1 课题背景

在高校校园中，二手物品交易具有天然的高频场景。教材、数码产品、生活用品、服饰配件等都具有较高的重复流转需求，但现有交易方式仍以班级群、朋友圈、校园墙或线下熟人转让为主。这类方式虽然门槛较低，但在信息组织、检索效率和交易范围控制方面存在明显缺陷，往往依赖人工沟通和碎片化传播，难以形成稳定、可持续的平台化服务。

随着微信小程序生态逐渐成熟，基于轻量化入口和移动端场景开发校园应用，已经成为一条比较常见也比较容易落地的路径。同时，位置服务技术可以借助终端定位、地图组件和空间距离计算，为“附近的人”“附近的商品”这类场景提供支持。对于校园二手交易来说，近距离交易不仅能减少沟通和面交成本，也更容易形成相对可信的交易关系。因此，将微信小程序与 LBS 技术结合起来做校园二手交易系统，是有现实依据的。

## 1.2 国内外研究现状

从国内应用现状来看，闲鱼、转转等平台已在二手交易领域形成成熟的商品发布、沟通和评价机制，但其业务覆盖范围广，面向全国用户，难以天然适配校园近场交易的快速匹配需求。高校内部虽然存在依托 QQ 群、微信群、论坛和校园表白墙进行二手转让的做法，但这些方案大多缺乏结构化管理、信用约束和就近检索机制，用户体验相对粗放。

# 第二章 相关理论与技术基础

## 2.1 LBS 技术概述

### 2.1.1 LBS 的定义

LBS，即 Location-Based Service，是指利用移动终端的位置信息，为用户提供与空间位置相关的服务。其核心流程通常包括定位信息获取、位置数据处理、业务规则计算以及结果展示。常见应用场景包括地图导航、周边搜索、同城推荐、地理围栏和轨迹分析等。

### 2.1.2 地理围栏与附近检索

在二手交易系统中，地理围栏可以理解为以用户当前位置为圆心、以设定半径为范围的虚拟空间边界。系统只返回边界内的商品数据，从而避免用户看到与自身位置距离过远、交易成本过高的商品。附近检索则是在地理围栏基础上进一步进行距离计算和排序，优先展示更接近用户的位置资源。

### 2.1.3 Haversine 距离计算原理

由于地球表面近似球体，直接用平面坐标计算距离会有一定误差，因此系统采用 Haversine 公式计算两点之间的大圆距离。其思路是先将经纬度转换为弧度，再利用球面三角关系计算两点间的弧长，最后乘以地球半径得到距离值。对本项目来说，这种方法实现不复杂，计算结果也能满足校园场景下的使用需求。

## 2.2 微信小程序与 Taro 技术

微信小程序是一种依托微信生态运行的轻量应用形态，具备无需下载安装、传播成本低、接入能力强等特点，非常适合校园服务类项目。其运行依托于微信客户端提供的宿主环境，能够直接调用定位、地图、网络请求、本地存储等能力。

Taro 是一个多端统一开发框架，允许开发者使用 React 语法编写小程序页面逻辑，并编译为微信小程序工程。相较于直接使用原生小程序语法，Taro 在组件复用、工程结构、类型支持和开发体验上具有优势。本文前端选择 Taro 3、React 18 与 TypeScript 组合，一方面便于管理页面状态和组件结构，另一方面也提升了代码可维护性。

## 2.3 Flask 与 RESTful API 技术

Flask 是 Python 生态中常用的轻量级 Web 框架，具有灵活、简单、可扩展的特点，适合中小型业务系统的快速构建。本文后端基于 Flask 实现业务接口，通过 Blueprint 方式组织用户、商品、订单、消息、评价、举报等不同模块的路由。

RESTful API 是一种资源导向的接口设计风格，其核心思想是通过统一的 URL 资源路径和 HTTP 方法表达不同业务动作。本文系统中，`GET` 用于查询数据，`POST` 用于创建资源，`PUT` 用于更新状态，`DELETE` 用于删除数据，接口响应以 JSON 为主，便于前端处理。

## 2.4 SQLAlchemy 与 MySQL

MySQL 是广泛应用于 Web 系统中的关系型数据库，具备稳定、成熟和易管理等优点。SQLAlchemy 作为 Python ORM 框架，能够将数据库表映射为 Python 类，使数据操作更贴近面向对象的编程方式。本文通过 SQLAlchemy 定义用户、商品、订单、消息、评价、举报等核心模型，并通过关系映射维护实体之间的关联。

## 2.5 JWT 与 WebSocket

JWT 用于解决前后端分离场景下的身份认证问题。用户登录成功后，服务端生成包含用户标识的令牌，前端后续请求通过请求头携带该令牌实现鉴权。相比传统 session 机制，JWT 更适合小程序与 API 服务的结合方式。

WebSocket 则为系统提供了持续的双向通信能力。本文中的聊天模块既支持基于 HTTP 的会话与消息查询，也设计了 WebSocket 消息推送服务，用于在用户在线时实现更接近实时的聊天体验。

## 2.6 本章小结

本章主要梳理了系统实现过程中用到的理论和技术，包括 LBS、微信小程序、Taro、Flask、SQLAlchemy、JWT 和 WebSocket。这些内容并不是孤立存在的，而是分别对应了项目中的不同部分，例如 LBS 负责附近检索，微信小程序与 Taro 负责页面展示和交互，Flask 与 MySQL 负责业务逻辑和数据存储，JWT 与 WebSocket 则补充了鉴权和即时通信能力。通过这些技术的配合，系统后续实现才有了较清晰的基础。

# 第三章 系统需求分析与总体设计

## 3.1 系统目标分析

本系统面向校园用户，目标是做一个可以直接在微信端使用的轻量化二手交易平台，满足用户发布商品、查找附近商品、沟通交易、完成订单和评价反馈等基本需求。除了用户端功能之外，系统也需要保留一定的平台治理能力，这样整体结构会更完整，也更符合毕业设计项目对平台型系统的要求。

从使用对象来看，系统主要包含两类角色：

1. 普通用户：完成登录、发布商品、浏览附近商品、聊天、下单、评价、收藏等操作。
2. 管理员：在后台管理系统中完成商品审核、举报处理、用户管理、分类管理、统计分析等操作。

## 3.2 功能需求分析

### 3.2.1 用户管理需求

系统应支持微信授权登录，保存用户昵称、头像、手机号和校区等信息，并允许用户修改个人资料。为了实现基本的社区信任机制，系统还需要维护用户信用分，并在个人中心展示发布数、收藏数和浏览历史等统计信息。

### 3.2.2 商品管理需求

用户应能够发布商品信息，内容包括标题、价格、分类、描述、图片和交易位置。系统应支持商品编辑、上下架、删除及我的发布列表查看。商品展示应支持分类筛选、关键词搜索和距离排序。

### 3.2.3 LBS 附近检索需求

系统应获取用户当前位置，根据商品发布位置进行距离计算，支持一定范围内商品的筛选和排序，并能在地图模式下显示商品地理分布。该模块是系统核心创新点之一，也是区别于普通二手列表系统的重要特征。

### 3.2.4 聊天与订单需求

用户在查看商品详情后，应能够直接联系卖家建立会话。系统需要支持会话列表、消息列表和消息发送功能。为了形成交易闭环，系统还应支持意向订单创建、订单完成确认和订单列表查看。

### 3.2.5 评价与举报需求

在订单完成后，买卖双方应能够互相评价。评价结果需要影响用户信用分，形成基本的信誉反馈机制。同时，系统应允许用户对商品、用户、消息或评价进行举报，为后台治理提供数据来源。

### 3.2.6 后台管理需求

根据后台管理系统 PRD，平台后续需要一套独立的 Web 管理端，至少支持管理员登录、工作台总览、用户管理、商品审核、举报处理、订单查看、分类维护和操作日志等能力。虽然该模块在当前代码仓库中尚处于扩展设计阶段，但从论文完整性和平台治理角度来看，应纳入系统总体设计范围。

## 3.3 非功能需求分析

### 3.3.1 性能需求

系统应保证常见业务在可接受时间内完成，例如首页商品列表加载、附近检索和聊天消息查询等。对于校园场景而言，单次附近检索返回数据规模通常不大，因此基于后端距离计算与排序的实现方式可以满足初期性能需求。

### 3.3.2 安全需求

系统需要保证用户身份认证可靠，防止未授权访问。对商品发布、评价提交、举报提交和订单操作等敏感动作应进行登录校验。后台管理端还需实现管理员独立认证和权限隔离。

### 3.3.3 可用性需求

系统界面应符合小程序交互习惯，操作流程清晰，错误提示明确。在网络异常、定位失败或图片上传失败等情况下，应给出友好的回退与提示机制。

### 3.3.4 可扩展性需求

系统需要为后续扩展后台管理、数据统计、推荐排序、商品审核、通知推送等功能预留接口和数据结构空间。现有数据库中已包含 `admin`、`report`、`notification` 和 `operation_log` 等表，为扩展平台治理能力奠定了基础。

## 3.4 系统总体架构设计

系统采用前后端分离的 B/S 架构。用户端基于微信小程序运行，前端负责页面展示、交互和接口调用；后端提供统一的 RESTful API 和 WebSocket 服务；MySQL 负责业务数据持久化；图片资源保存在文件系统或上传目录中。

系统整体可划分为四层：

1. 表现层：Taro + React 小程序页面，以及后续扩展的 Vue3 后台管理端。
2. 业务层：Flask 后端业务服务，负责用户、商品、订单、消息等逻辑。
3. 数据访问层：SQLAlchemy ORM。
4. 数据存储层：MySQL 和上传文件目录。

## 4.4 LBS 核心算法实现

### 4.4.1 距离计算实现

后端在商品模块中实现了 `haversine_distance` 方法，对用户位置和商品位置进行球面距离计算。其处理流程如下：

1. 将用户纬度、经度以及商品纬度、经度统一转换为弧度。
2. 计算纬度差与经度差。
3. 使用 Haversine 公式求得球面弧长。
4. 乘以地球平均半径，得到公里级距离。

### 4.4.2 附近检索实现

在 `nearby` 接口与商品列表接口中，系统会根据用户传入的 `lat`、`lng` 和 `radius` 参数对商品进行筛选。处理逻辑包括：

1. 查询状态为上架且审核通过的商品。
2. 对每个商品计算与用户之间的距离。
3. 若设置了半径限制，则过滤超出范围的商品。
4. 对剩余商品按距离升序排序。
5. 将距离信息写入返回结果，供前端展示。

该算法虽然没有引入数据库空间索引或 GIS 引擎，但在校园场景下数据量通常不会太大，用这种方式实现成本较低，结果也比较直观，适合当前项目阶段使用。

# 第六章 总结与展望

## 6.1 工作总结

本文围绕校园场景下的二手交易需求，设计并实现了一套基于 LBS 的微信小程序校园二手交易系统。技术上，前端采用 Taro + React + TypeScript，后端采用 Flask + SQLAlchemy + MySQL，并通过 JWT 处理用户认证，以 WebSocket 作为即时通信扩展方案。业务上，系统已经实现了用户管理、商品发布、附近检索、聊天、订单、评价、信用分、举报、收藏和浏览历史等主要功能，整体交易流程已经基本串联起来。

在系统特色方面，项目重点实现了基于 Haversine 公式的附近检索逻辑，将用户当前位置与商品发布位置结合起来，通过距离计算、半径过滤和距离排序提升校园近场交易的效率。和只按发布时间展示商品的普通二手列表相比，这种方式更符合校园线下面交的实际场景。

在系统完整性方面，论文还补充了后台管理系统的扩展设计，把管理员登录、商品审核、举报处理、用户封禁、分类管理和操作日志等平台治理功能纳入了整体方案。这一部分虽然还需要继续落地，但至少把平台化应用应有的管理侧思路补上了。

## 6.2 创新点与特色

这个项目的主要特色可以概括为以下几点：

1. 将 LBS 技术与校园二手交易场景深度结合，突出附近检索与本地化交易。
2. 采用微信小程序作为用户端入口，降低了使用门槛与传播成本。
3. 建立了商品、聊天、订单、评价、信用分相结合的交易闭环。
4. 在移动端系统之外，补充后台管理系统扩展设计，提高平台治理能力与毕设完整度。

## 6.3 存在的不足

虽然系统已经完成了主要业务功能，但从实际项目角度看，仍然存在一些不足：

1. 当前商品附近检索主要通过后端遍历计算实现，面对大规模数据时性能有限。
2. 商品审核和后台管理系统目前仍以扩展设计为主，尚需进一步完整实现。
3. 即时通信虽然设计了 WebSocket 服务，但在复杂网络环境下仍需更充分测试。
4. 评价与信用分模型较为简单，尚未引入更复杂的风控与信誉评估机制。
5. 系统尚未对支付、退款、仲裁等更复杂交易环节进行深入实现。

## 6.4 后续展望

围绕上述不足，系统后续可从以下方向进一步优化：

1. 引入 GIS 或空间索引能力，提升附近检索性能。
2. 完整实现后台管理系统，并接入商品审核与举报联动处置。
3. 丰富推荐逻辑，在距离排序基础上结合浏览量、收藏量和发布时间进行综合推荐。
4. 加强聊天模块能力，支持图片消息、已读回执和断线重连。
5. 构建更完善的信用分与风控体系，提高平台治理水平。
