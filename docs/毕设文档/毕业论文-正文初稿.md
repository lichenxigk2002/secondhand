# 基于 LBS 的校园二手交易微信小程序的设计与实现

## 摘要

高校学生在教材资料、数码产品、生活用品、服饰配件等物品上存在较为频繁的闲置流转需求。课程结束、搬迁宿舍、毕业离校和设备更新等场景都会产生大量可再次利用的物品。传统校园二手交易主要依赖微信群、QQ群、朋友圈、校园墙或熟人转介绍完成，虽然发布门槛较低，但商品信息分散、检索效率不高，交易沟通、订单确认、评价反馈和违规处理难以在统一平台内沉淀。与此同时，校园二手交易具有明显的空间属性，买卖双方通常更倾向于在宿舍区、教学楼、食堂、校门等熟悉地点完成线下面交。因此，如何结合位置服务提升商品发现效率，并形成轻量、可追踪、可治理的校园二手交易流程，具有较强的研究与实践意义。

本文面向高校近场交易场景，设计并实现了一套基于 LBS 的校园二手交易微信小程序。系统采用前后端分离架构，用户端使用 Taro、React 18 与 TypeScript 开发，后端采用 Flask、SQLAlchemy 与 MySQL 构建业务接口，并使用 JWT 完成身份认证。系统同时引入 WebSocket 支撑聊天消息实时推送，构建 Vue 3 后台管理端支撑平台治理。系统核心围绕“商品发布、附近发现、沟通交易、确认完成、评价反馈、后台治理”展开，实现了用户登录与资料维护、商品发布与编辑、图片上传、位置选择、首页检索、附近排序、地图展示、商品收藏、浏览历史、商品留言、聊天会话、意向订单、交易完成、评价信用、举报反馈、分类维护、商品审核、用户管理和操作日志等功能。

本文重点完成了三方面工作。第一，结合校园线下面交需求设计 LBS 商品检索流程，在后端基于 Haversine 公式计算用户与商品位置之间的球面距离，并在商品列表和附近检索接口中支持半径过滤与距离排序。第二，构建轻量交易闭环，将商品详情、聊天会话、意向订单、交易完成、评价提交和信用分更新串联起来，使系统不仅停留在商品展示层面，还能够记录交易过程。第三，补充后台管理与平台治理能力，使商品审核、举报处理、用户状态管理、评价管理和操作日志能够支撑系统的规范运行。测试与构建验证表明，系统能够完成毕业设计范围内的主要业务流程，具备较好的完整性、可演示性和实际应用参考价值。

**关键词：** LBS；微信小程序；校园二手交易；Taro；Flask；Haversine

## Abstract

College students frequently exchange idle goods such as textbooks, digital devices, daily necessities, and clothing. A large number of reusable items are generated during course completion, dormitory relocation, graduation, and device replacement. Traditional campus second-hand trading mainly relies on social groups, moments, campus bulletin channels, or acquaintances. These methods are easy to use, but product information is scattered, retrieval efficiency is limited, and communication, transaction confirmation, evaluation, and governance are difficult to manage in a unified platform. Campus second-hand trading also has strong spatial characteristics, since buyers and sellers usually prefer face-to-face transactions in familiar areas such as dormitories, teaching buildings, canteens, and campus gates. Therefore, it is meaningful to introduce location-based services into campus second-hand trading and build a lightweight, traceable, and governable transaction process.

This thesis designs and implements an LBS-based campus second-hand trading WeChat Mini Program. The system adopts a front-end and back-end separated architecture. The mobile client is developed with Taro, React 18, and TypeScript. The back end is implemented with Flask, SQLAlchemy, and MySQL, while JWT is used for identity authentication. WebSocket is introduced for real-time chat message delivery, and a Vue 3 administration console is implemented for platform governance. The core business process is organized around goods publishing, nearby discovery, communication, transaction confirmation, evaluation feedback, and administration. The system implements user login and profile management, goods publishing and editing, image upload, location selection, homepage retrieval, nearby sorting, map display, favorites, browsing history, goods comments, chat conversations, intention orders, transaction completion, evaluation and credit score update, report submission, category maintenance, goods auditing, user management, and operation logs.

The main work of this thesis includes three aspects. First, an LBS-based goods retrieval process is designed for campus face-to-face trading. The back end calculates the spherical distance between the user location and goods location with the Haversine formula, and supports radius filtering and distance sorting in goods list and nearby search interfaces. Second, a lightweight transaction loop is built by connecting goods details, chat conversations, intention orders, transaction completion, evaluation submission, and credit score updates. Third, administration and governance capabilities are provided so that goods auditing, report handling, user status management, evaluation management, and operation logs can support platform operation. The testing and build verification show that the system can support the main business processes required by the graduation project and has practical demonstration value.

**Key Words:** LBS; WeChat Mini Program; campus second-hand trading; Taro; Flask; Haversine

## 第1章 绪论

### 1.1 研究背景

校园二手交易是高校生活中较为常见的交易场景。学生在不同学习阶段会持续产生闲置物品，例如课程结束后的教材资料、设备更新后的数码产品、宿舍整理后的生活用品，以及毕业离校时集中转让的家具、收纳用品和服饰配件等。这些物品通常仍具备使用价值，如果能够在校园内部高效流通，不仅可以降低学生的生活和学习成本，也有助于减少资源浪费，符合绿色校园和循环利用的理念。

与面向全社会的综合二手交易平台相比，校园二手交易具有更强的近场特征。买卖双方往往位于同一所高校或相邻校区，交易距离较短，线下面交更方便；双方对校园环境较熟悉，能够通过宿舍楼、教学楼、图书馆、食堂、校门等地点快速约定见面位置；商品类型也更集中，教材资料、电子设备和宿舍用品等类别具有明显的校园属性。因此，校园交易并不只是一般 C2C 二手交易的缩小版，而是一个更加依赖位置、即时沟通和社区信任的具体应用场景。

在传统模式下，校园二手信息常分散在微信群、QQ群、朋友圈、校园墙或熟人关系中。这类方式具有轻量、直接、传播速度快等优点，但也存在明显不足。首先，商品信息通常以文本和图片混合形式发布，缺少统一的标题、价格、分类、成色、位置和状态字段，后续检索与筛选效率较低。其次，信息流主要按发布时间滚动，旧商品容易被新消息覆盖，买家需要反复翻找才能找到目标商品。再次，聊天、成交、评价和举报分散在不同工具中，交易过程难以形成完整记录。最后，管理员或运营者难以对虚假商品、恶意评价、违规内容和纠纷线索进行统一处理。

微信小程序具有无需安装、入口轻、传播便利和用户使用门槛低等特点，适合作为校园服务类应用的承载形式。小程序能够调用定位、地图、相册、文件上传、网络请求和本地存储等能力，可以较好地满足校园二手交易中商品发布、图片展示、位置选择、附近检索和交易沟通等需求。LBS 技术则能够将用户位置与商品位置结合起来，通过距离计算、半径过滤、地图展示和距离排序增强商品发现效率。将微信小程序与 LBS 技术结合，可以使校园二手交易从单纯“按发布时间浏览”扩展为“按附近范围发现”，更贴近线下面交需求。

基于上述背景，本文设计并实现一套基于 LBS 的校园二手交易微信小程序。系统以校园近场交易为切入点，在用户端提供商品发布、附近浏览、地图展示、聊天沟通、订单确认和评价反馈等功能，在后台端提供商品审核、用户管理、举报处理和操作日志等治理功能，力求形成一个结构完整、流程清晰、可运行可演示的毕业设计系统。

### 1.2 国内外研究现状

#### 1.2.1 二手交易平台发展现状

国内二手交易平台已经形成较成熟的产品形态，典型平台通常围绕商品发布、搜索推荐、在线沟通、信用评价、担保交易和纠纷处理等能力进行建设。这些平台面向社会大众，商品数量大、品类丰富，能够满足跨城市或同城范围内的二手交易需求。然而，校园场景具有用户群体相对集中、交易半径较小、见面地点明确和商品类型较固定等特点，大型平台的广域流量优势在校园内部并不一定能够完全转化为匹配效率。对于学生用户而言，能否快速找到距离较近、可当面验货、沟通成本较低的商品，往往比平台规模本身更加重要。

在高校内部，许多学校已形成基于微信群、QQ群、公众号推文、校园墙和论坛帖子的二手交易习惯。这类方式依托已有社交关系，具备一定社区信任基础，但也存在结构化程度不足的问题。商品发布往往没有统一模板，价格、成色、交易地点和联系方式等信息容易遗漏；买家查找商品时主要依靠关键词搜索或人工翻阅，效率不稳定；商品是否售出、是否被举报、是否存在纠纷，通常缺少统一状态记录。因此，校园二手交易系统需要在保持轻量使用体验的同时，引入结构化商品管理和平台治理机制。

国外二手交易平台较早重视位置筛选能力，例如面向社区或本地交易的平台通常支持按距离、城市或区域筛选附近商品。位置服务能够降低物流成本，提高线下面交可行性，也有助于增强买卖双方对交易环境的熟悉感。但从公开研究和常见产品实践看，专门面向高校封闭或半封闭社区的 LBS 二手交易系统仍有进一步研究空间，尤其是在小程序形态、校园区域筛选、附近检索、轻量订单和后台治理结合方面，仍适合通过毕业设计进行系统化实践。

#### 1.2.2 LBS 技术在移动应用中的应用

LBS（Location Based Service，基于位置的服务）已广泛应用于导航、外卖、打车、同城服务、门店推荐和社交应用中。其基本流程通常包括获取用户位置、存储目标对象位置、计算空间距离、进行范围过滤或排序，并在地图或列表中展示结果。在校园二手交易中，LBS 的价值主要体现在三个方面：一是帮助买家快速识别商品与自己的距离，减少无效沟通；二是帮助卖家选择较明确的交易地点，提升面交效率；三是通过校园范围内的空间约束，增强交易场景的真实性和可控性。

微信小程序提供了定位获取、位置选择和地图展示等能力，开发者可以通过相关接口获取用户当前经纬度，也可以让用户在地图上选择商品交易地点。后端服务接收到经纬度后，可以通过 Haversine 公式或数据库空间函数计算两点之间的距离。对于校园范围内的数据规模，应用层距离计算具有实现简单、可解释性强和便于演示的特点；当数据量进一步扩大时，也可以引入空间索引、地理哈希或数据库 GIS 能力进行优化。

现有研究中，LBS 更多用于出行、外卖、门店推荐等强位置依赖场景，而在二手交易尤其是校园二手交易中的应用仍有扩展空间。校园交易对定位精度的要求不一定达到导航级别，但更关注距离排序、地点可理解性和隐私边界。因此，系统设计时需要在检索效率、用户体验和位置隐私之间取得平衡，避免过度暴露用户实时位置，同时保证商品位置能够满足面交判断需求。

#### 1.2.3 微信小程序在校园场景中的实践

微信小程序已广泛应用于高校校园服务场景，包括课表查询、校园卡充值、通知公告、活动报名、失物招领、报修服务和校园导览等。其优势在于用户无需下载安装独立 App，能够通过微信入口快速访问，也便于在班级群、学院群和社团群中传播。对于校园二手交易而言，小程序可以降低首次使用成本，使用户在需要出售或购买闲置物品时快速进入系统。

从技术角度看，微信小程序与前后端分离架构结合较为成熟。前端可以使用原生小程序框架，也可以使用 Taro 等跨端框架提升组件复用能力；后端可以采用 Flask、Spring Boot、Go 等框架提供 RESTful API；数据库可使用 MySQL 保存结构化业务数据；聊天消息可以通过 WebSocket 提升实时性。本文采用 Taro、React、Flask、MySQL 和 WebSocket 组合，既能满足系统功能需求，也便于在毕业设计中展示完整的软件工程实现过程。

### 1.3 课题目的与意义

#### 1.3.1 课题目的

本文的研究目标是设计并实现一个适用于高校校园场景的二手交易微信小程序，使学生用户能够在移动端完成闲置商品发布、附近商品浏览、商品详情查看、收藏留言、聊天沟通、意向订单创建、交易完成确认和评价反馈等操作，并使管理员能够通过后台管理端对用户、商品、举报、评价和日志进行统一管理。

具体而言，本文希望解决以下问题。第一，分析校园二手交易在位置、沟通、交易确认和平台治理方面的需求，明确系统功能需求与非功能需求。第二，构建完整的系统架构，完成小程序用户端、Flask 后端、MySQL 数据库、WebSocket 聊天服务和 Vue 后台管理端的协同设计。第三，实现基于 LBS 的附近检索能力，使用 Haversine 公式计算用户与商品之间的距离，并支持半径过滤和距离排序。第四，实现轻量交易闭环，使系统能够从商品展示延伸到聊天、订单、完成确认、评价和信用分更新。第五，通过功能测试和构建验证检查系统主要流程的可用性，为毕业设计答辩和后续优化提供依据。

#### 1.3.2 课题意义

从理论意义看，本文将 LBS 技术与校园二手交易场景结合，对位置获取、距离计算、附近检索、地图展示和交易闭环之间的关系进行了系统梳理。系统采用 Haversine 公式完成距离计算，并结合商品状态、审核状态、关键词、分类和校内区域等条件进行综合检索，为校园近场交易系统的设计提供了一个可参考的实现样本。

从实践意义看，系统能够改善传统校园二手交易中信息分散、检索困难、状态不清和治理不足的问题。学生用户可以通过小程序快速发布闲置商品，也可以根据当前位置发现附近商品，降低沟通与面交成本。管理员则可以通过后台端处理举报、审核商品和查看操作日志，提高平台运行的规范性。对于开发者而言，本课题覆盖前端页面、后端接口、数据库建模、身份认证、即时通信、位置服务和后台管理等多个环节，能够较完整地体现软件工程综合实践能力。

7. 补充 AI 发布辅助和发布前预检能力，提高商品标题、描述、分类和价格信息的完整度。
8. 设计功能测试用例并完成构建验证，检查系统主要流程的可用性和演示稳定性。

本文技术路线为：首先调研校园二手交易、微信小程序、LBS 和二手交易平台相关技术；其次完成需求分析和数据库设计；随后分别实现小程序用户端、后端 API、WebSocket 服务和后台管理端；在核心算法方面实现 Haversine 距离计算和附近排序；最后通过功能测试、典型流程测试和构建验证评估系统效果。

## 第2章 相关技术与理论基础

### 2.1 微信小程序与 Taro

微信小程序是一种运行在微信生态内的轻量应用形态，用户无需下载安装独立客户端即可使用。小程序提供页面路由、组件渲染、网络请求、本地缓存、相册选择、文件上传、定位获取和地图展示等能力，适合承载校园服务类应用。对于校园二手交易系统而言，小程序能够满足商品列表浏览、商品发布、图片上传、位置选择、地图展示、聊天入口和个人中心等基础需求。

Taro 是一个多端统一开发框架，支持使用 React、Vue 等前端技术开发微信小程序及其他端应用。本文用户端采用 Taro 3、React 18 与 TypeScript 开发。React 的组件化机制适合拆分商品卡片、搜索筛选栏、发布表单、聊天气泡、订单卡片和底部导航等界面元素；TypeScript 能够约束接口返回结构和页面状态类型，提高代码可维护性。Taro 的跨端抽象还降低了小程序 API 与普通前端组件之间的衔接成本，使开发过程更加统一。

### 2.2 Flask、SQLAlchemy 与 MySQL

Flask 是 Python 生态中常用的轻量级 Web 框架，具有结构简洁、扩展灵活和学习成本较低等特点。本文后端采用 Flask 构建 RESTful API，并按照 Blueprint 对用户、商品、上传、收藏、浏览、消息、订单、评价、举报、后台管理和 AI 辅助等模块进行路由拆分，使不同业务逻辑保持相对独立。

SQLAlchemy 用于完成对象关系映射，将数据库表映射为 Python 模型类。系统通过模型类描述用户、分类、商品、订单、会话、消息、评价、信用分日志、举报、收藏、浏览历史、商品留言、管理员和操作日志等实体。ORM 方式能够降低直接编写 SQL 的复杂度，也便于在模型层统一维护字段和关联关系。

MySQL 作为系统主要数据存储，用于持久化保存结构化业务数据。商品图片通过上传接口保存到后端上传目录，并将可访问 URL 写入商品图片字段。对于毕业设计系统而言，MySQL 能够满足数据可靠性、查询能力和部署便利性要求。

### 2.3 JWT 身份认证

系统用户端和后台端均采用 JWT（JSON Web Token）进行身份认证。普通用户登录成功后，后端签发包含用户 ID、过期时间等信息的令牌，前端将令牌保存到本地存储中，后续请求通过 `Authorization` 请求头携带该令牌。后端在需要登录的接口中解析令牌并获取当前用户，从而控制商品发布、收藏、聊天、订单、评价和举报等操作。

后台管理员使用独立登录接口。管理员令牌中包含 `token_type=admin` 标识，用于与普通用户令牌区分。后台管理接口通过管理员鉴权装饰器控制访问，避免普通用户令牌访问管理端功能。该设计实现简单，能够满足毕业设计阶段的身份识别和访问控制要求。

### 2.4 WebSocket 即时通信

校园二手交易中，买卖双方往往需要就商品成色、价格、配件、面交地点和时间进行沟通。若完全依赖普通 HTTP 轮询，消息实时性较弱，且会产生额外请求开销。因此，系统在 HTTP 消息接口之外提供 WebSocket 服务，用于在线消息推送。

用户进入聊天页后，小程序携带 JWT 建立 WebSocket 连接；服务端验证令牌后保存在线连接。用户发送消息时，服务端将消息写入数据库，更新会话最后消息和未读数，并向会话双方在线连接推送消息。同时，系统仍保留 HTTP 消息发送和历史消息查询接口。这样即使 WebSocket 连接异常，用户仍可以通过 HTTP 接口完成消息发送和历史记录加载，增强聊天模块的稳定性。

### 2.5 LBS 与 Haversine 距离计算

LBS 即基于位置的服务，通常包括定位获取、位置存储、空间计算、范围过滤和结果展示等环节。在本系统中，用户端通过微信小程序定位能力获取当前位置，商品发布时通过位置选择记录交易地点，后端则根据用户当前位置和商品位置计算距离。

本文采用 Haversine 公式计算两点之间的球面距离。设用户当前位置为 `(lat1, lng1)`，商品位置为 `(lat2, lng2)`，经纬度在计算前转换为弧度，地球平均半径取 `6371km`，则距离计算可表示为：

```text
a = sin²((lat2 - lat1) / 2)
    + cos(lat1) × cos(lat2) × sin²((lng2 - lng1) / 2)

d = 2R × arcsin(√a)
```

其中 `d` 表示两点之间的球面距离。由于校园范围通常较小，Haversine 公式的精度能够满足商品距离展示、附近排序和半径过滤需求。相较于复杂空间索引，该方法实现直观、便于解释，适合毕业设计系统的算法展示。

### 3.3 非功能需求分析

系统在非功能方面主要关注可用性、可维护性、安全性、性能和演示稳定性。

可用性方面，小程序页面需要提供清晰的空状态、错误提示和登录引导。定位失败时，系统应回退到普通商品列表，避免用户因拒绝定位或定位异常而无法浏览商品。商品发布表单需要进行必要校验，减少标题、价格、图片和位置缺失造成的无效数据。

可维护性方面，前端按页面、组件和服务接口拆分，后端按业务蓝图拆分，避免所有逻辑集中在单一文件中。数据库模型围绕用户、商品、交易、消息、评价和治理建立清晰实体关系，便于后续扩展。

安全性方面，用户端敏感操作需要登录校验，后台接口需要管理员令牌。普通用户只能管理自己发布的商品和相关订单，管理员接口与用户接口相互隔离。对于位置数据，系统保存的是商品交易地点和用户请求时提交的位置参数，不在后台长期展示用户实时轨迹。

性能方面，系统当前面向校园中小规模数据，附近检索采用应用层距离计算即可满足演示和基本使用需求。若未来数据量扩大，可进一步引入经纬度范围预过滤、地理哈希或数据库空间索引。

演示稳定性方面，系统需支持局域网 IP 访问，后端提供健康检查接口，聊天模块同时支持 WebSocket 与 HTTP 回退，避免单一通信方式异常导致演示中断。

### 3.4 系统总体架构设计

系统采用前后端分离架构，并在用户端之外提供独立后台管理端。整体可分为四层。

第一层为表现层，包括微信小程序用户端和 Vue 后台管理端。小程序承担商品交易流程，后台管理端承担平台治理流程。

第二层为接口服务层，包括 Flask RESTful API 和独立 WebSocket 聊天服务。Flask 服务负责大部分业务接口，WebSocket 服务负责聊天实时推送。

第三层为数据访问层，使用 SQLAlchemy ORM 进行数据库操作，降低直接编写 SQL 的复杂度。

第四层为数据存储层，包括 MySQL 数据库和本地图片上传目录。MySQL 保存用户、商品、订单、消息、评价、举报和后台日志等数据，图片上传目录保存商品图片文件。

## 第4章 系统详细设计与实现

### 4.1 用户登录与资料模块实现

用户登录通过小程序 `Taro.login` 获取临时 code，再提交给后端登录接口。后端在配置了微信 AppID 和 Secret 时调用微信接口换取 openid；在开发调试环境中，为便于本地演示，也支持根据 code 生成开发态 openid。用户首次登录时后端创建用户记录，随后签发 JWT 返回给前端。

前端将 token 和用户资料保存到本地存储中，后续请求通过统一请求封装自动携带 `Authorization` 请求头。个人中心页面会在进入时加载用户资料、未读消息数和统计数据，包括我的发布数、收藏数和浏览历史数。资料编辑页支持修改昵称、手机号和校内区域信息，该字段同时用于首页的区域筛选。

该模块的重点是保证登录态在小程序多个页面之间保持一致。当前端检测到接口返回未登录或令牌失效时，会引导用户重新登录；当用户修改资料后，前端会及时刷新本地用户信息，避免个人中心显示旧数据。

从实现流程看，用户模块并不直接暴露 openid 等平台身份字段给前端，而是由后端负责维护微信身份与系统用户之间的映射关系。前端只关心登录结果、用户资料和 token。这样可以降低小程序端对后端用户模型的依赖，也便于后续扩展手机号绑定、头像更新和账号状态校验等能力。对于被后台禁用的用户，后端可以在鉴权阶段直接拦截其敏感操作，从而保证用户状态管理能够在各业务模块中统一生效。

### 4.2 商品发布与商品管理实现

商品发布页要求用户填写标题、价格、分类、描述、图片和位置。前端对标题长度、价格合法性、图片数量、位置和描述进行基础校验，避免明显不完整的商品进入系统。图片通过上传接口保存到后端上传目录，后端返回可访问 URL。位置通过小程序位置选择能力获取经纬度，用于后续附近检索。

用户发布成功后，商品会保存到 `goods` 表中。当前毕业设计演示版本中，商品默认设置为审核通过，便于发布后立即在首页展示；同时系统仍保留 `audit_status` 字段和后台审核接口，用于展示平台治理能力。我的发布页支持查看自己发布的商品，并对商品进行编辑、上下架和删除。当商品已经关联订单时，后端限制直接删除，以避免交易数据异常。

在商品管理方面，系统区分商品业务状态与审核状态。业务状态用于表示商品在售、下架或已售，审核状态用于表示是否通过后台审核。两类状态分离后，系统既能支持用户自主上下架，也能支持管理员从平台治理角度进行审核和下架处理。

商品发布接口在后端需要完成三类校验。第一类是字段完整性校验，包括标题、价格、分类、描述和位置等关键字段是否存在；第二类是业务合法性校验，例如价格不能为负数，商品分类需要存在，商品所有者必须为当前登录用户；第三类是状态约束校验，例如已经完成交易的商品不能被普通用户随意删除。通过前后端双重校验，可以减少异常数据写入数据库的概率，也能提升系统在答辩演示时的稳定性。

### 4.3 首页检索与 LBS 实现

首页是系统的主要入口，负责商品发现。页面进入后优先请求用户定位，定位成功时将 `lat`、`lng` 和默认半径提交给商品列表接口；定位失败时则回退为普通商品列表。用户还可以通过关键词、分类和校内区域进行筛选，并在列表模式和地图模式之间切换。

后端商品列表接口在接收到坐标参数后，会对查询出的商品计算距离，并将距离写入返回结果。附近检索接口则要求必须传入坐标，并对上架且审核通过的商品逐一计算距离，过滤半径外商品，再按距离升序排序。前端根据返回的 `distance` 字段展示“距离 xx km”的提示，使用户能够快速判断商品是否适合面交。

附近检索的核心过程可以概括为：接收用户当前位置和检索半径；查询符合状态、审核、关键词、分类和校内区域条件的商品；读取每个商品的经纬度；使用 Haversine 公式计算距离；过滤半径之外的商品；按照距离升序排序并返回结果。该实现没有使用复杂的数据库空间索引，而是在应用层完成距离计算。对于毕业设计和校园中小规模演示数据而言，这种方案代码清晰、易于说明，也便于在论文中展示算法过程。

地图展示用于补充列表模式。列表模式适合快速比较价格、标题和距离；地图模式适合观察商品在校园内的大致分布。用户可以根据自己的使用习惯在两种展示方式之间切换。

在距离计算结果展示方面，系统不会直接暴露过于精确的实时用户位置，而是以商品与用户当前位置之间的距离作为辅助信息返回前端。对于距离较近的商品，前端展示公里数或米级近似值即可满足线下面交判断需求。该处理既体现了 LBS 的核心价值，也避免将位置服务设计成对用户轨迹的持续追踪。若后续上线到真实校园环境，还可以进一步加入位置授权说明、默认模糊化展示和用户主动关闭定位等隐私保护措施。

### 4.4 商品详情、收藏、浏览历史与留言实现

商品详情页用于承接首页点击后的进一步交易动作。页面展示商品图片、价格、标题、状态、审核状态、浏览量、分类、卖家资料、信用分、商品描述和交易地点。用户进入详情后，前端会调用浏览记录接口，后端增加商品浏览量并写入浏览历史。登录用户可以收藏或取消收藏商品，也可以在个人中心查看收藏列表和浏览历史列表。

详情页还支持商品留言功能。留言与聊天不同，留言是围绕商品的公开评论，用于补充询问成色、配件、保修、价格或交易地点等信息。后端为留言提供独立接口，并按创建时间倒序返回列表。留言功能能够让部分通用问题在详情页中沉淀，减少卖家重复回复。

举报入口也放在详情页中，用户可以选择虚假信息、违规内容、欺诈行为等原因提交举报。举报记录进入后台管理端后，管理员可以查看举报原因、补充说明、目标对象和处理状态，并填写处理备注。该设计使用户端反馈与后台治理形成闭环。

收藏、浏览历史和留言三个功能虽然不是交易主链路的强制步骤，但能够明显改善商品发现体验。收藏功能帮助用户暂存感兴趣商品，避免反复搜索；浏览历史用于记录用户已经查看过的商品，便于回溯；留言功能则适合承载公开问题，减少卖家重复解释。三者共同增强了商品详情页的信息沉淀能力，使系统从单纯的商品列表展示扩展为具有互动记录的交易平台。

### 4.5 聊天、订单与评价闭环实现

交易闭环是本文实现的重点之一。用户在商品详情页点击“联系卖家”后，小程序会进入聊天页。聊天页首先调用获取或创建会话接口，后端根据当前用户和卖家用户确定双方 ID，若会话不存在则创建会话。若本次聊天关联了商品 ID，前端会继续调用订单创建接口，后端创建该商品的意向订单。

消息发送支持 WebSocket 和 HTTP 两条路径。WebSocket 在线时，消息通过长连接发送到服务端，服务端写入数据库后广播给会话双方；当 WebSocket 不可用时，前端可回退到 HTTP 消息发送接口。会话列表根据最后消息时间排序，并展示未读数和最后一条消息。历史消息通过分页接口加载，避免一次性读取过多数据。

订单页展示买家和卖家相关订单。交易完成后，任一交易方可确认完成，后端将订单状态更新为已完成，同时将商品状态更新为已售，并取消同一商品的其他进行中意向订单。订单完成后，用户可以进入评价页，对交易对象提交星级和文字评价。评价提交后，后端以 3 星为基准计算信用分变化，将结果写入用户表和信用分日志表。

该闭环使系统能够记录从“产生交易意向”到“完成交易反馈”的过程。相较于只提供商品展示和联系方式的简单平台，聊天、订单和评价模块能够增强交易可追踪性，也便于后续扩展纠纷处理和信用评价规则。

订单创建采用“意向订单”方式，而不是在用户第一次联系卖家时直接视为正式成交。这样更符合校园二手交易的实际过程：买家先通过聊天了解商品情况，双方再线下验货和议价，最终确认是否完成交易。系统通过订单状态记录交易阶段，避免聊天记录和交易结果完全脱节。商品被确认售出后，系统会将同一商品的其他进行中意向订单取消，防止多个买家同时认为自己已经锁定商品。

评价模块与信用分模块相互关联。评价本身保存星级、文字内容、评价人和被评价人等信息，信用分日志则记录本次评价导致的分值变化。采用日志表记录信用分变化，可以避免只保存最终分数而无法追溯来源的问题。后续若需要调整信用分规则，也可以基于历史日志进行分析，而不是重新推断每一次评分的影响。

### 4.6 后台管理端实现

后台管理端是系统治理能力的体现。管理员登录后进入工作台，可查看用户数、商品数、待审核商品数、待处理举报数、订单数和已完成订单数，并查看近 7 日趋势数据。用户管理页面支持搜索用户并修改用户状态；商品管理页面支持查看商品、审核商品和强制下架；举报管理页面支持查看举报原因、补充说明并填写处理备注。

后台还提供分类管理、订单查看、评价管理、管理员账号管理和操作日志查询。分类管理用于维护商品分类；订单查看用于辅助核查交易状态；评价管理支持删除异常评价；账号管理支持新增和更新管理员；操作日志记录管理员关键动作，便于答辩时说明系统具备审计能力。

后台管理端与用户端共享同一后端服务，但使用独立的管理员鉴权机制。这样既能复用业务数据，又能避免普通用户访问管理接口。对于毕业设计而言，后台管理端能够体现系统不仅关注用户使用流程，也考虑了平台运行和内容治理。

后台管理端的实现重点不在于复杂视觉效果，而在于管理流程清晰。工作台页面用于快速观察平台运行状态，用户管理页面用于处理异常账号，商品管理页面用于审核和下架商品，举报管理页面用于承接用户反馈，操作日志用于记录关键管理行为。通过这些模块，系统能够从“用户可用”进一步扩展为“平台可管”，这也是交易类系统与普通信息展示系统之间的重要区别。

### 4.7 AI 发布辅助实现

AI 发布辅助不是系统交易主链路的必需功能，但可以提升商品发布体验。后端提供商品草稿生成和发布前预检两个接口。商品草稿生成接口根据用户已填写的标题、描述、分类、图片数量和位置名称生成更规范的标题、描述、分类和价格建议；发布前预检接口检查标题长度、图片数量、价格格式、描述完整度和敏感交易词，并返回风险等级、缺失字段和建议。

前端发布页允许用户将 AI 建议按字段应用到表单中，也可以在最终提交前执行预检。如果 AI 服务不可用，后端还提供本地兜底规则，保证演示阶段不会因为外部模型不可用导致发布流程中断。

AI 发布辅助的定位是增强信息规范性，而不是替代用户决策。最终发布内容仍由用户确认，系统只提供标题优化、描述补全、分类建议和风险提示，从而降低不完整商品信息进入平台的概率。

### 4.8 部署与运行实现

### 5.3 典型流程测试方案

典型流程测试使用两个普通用户账号和一个管理员账号完成。用户 A 发布一件带图片和位置的商品，用户 B 在首页通过附近检索找到该商品，进入详情页后收藏商品、查看卖家信用分并联系卖家。进入聊天页后，系统创建或复用会话，并创建意向订单。双方完成线下面交后，用户 B 或用户 A 在订单页确认完成，系统将订单状态更新为已完成，并将商品标记为已售。随后用户进入评价页提交评分和文字评价，系统更新被评价用户信用分。若用户发现商品或评价异常，可以提交举报，管理员在后台查看并处理。

该流程覆盖了系统最核心的业务链路，也适合作为毕业答辩演示主线。答辩时可先展示用户端发布和附近检索，再展示聊天订单评价，最后切换到后台管理端说明平台治理能力。

在实际测试过程中，需要特别关注数据状态是否能够在多个页面之间正确同步。例如商品发布成功后，首页列表、地图标记和“我的发布”页面都应能够看到该商品；订单确认完成后，订单列表中的状态和商品详情中的状态都应发生变化；评价提交后，被评价用户的信用分和评价列表应能够体现本次评价结果。如果某个页面仍显示旧状态，说明前端刷新逻辑或后端返回数据存在问题，需要进一步排查。

典型流程测试还验证了系统的异常处理能力。当定位授权失败时，首页能够回退到普通列表，不影响用户继续浏览商品；当 WebSocket 连接不可用时，聊天历史仍可通过 HTTP 接口加载；当用户试图删除已关联订单的商品时，后端会限制该操作，避免破坏交易数据。这些处理虽然不一定直接体现在页面截图中，但对系统演示稳定性具有重要意义。

### 5.4 LBS 与交易闭环测试分析

LBS 附近检索是系统区别于普通二手商品列表的核心能力，因此测试时需要单独验证位置参数、距离字段和排序结果。测试方法为准备多条带有不同经纬度的商品数据，在小程序端授权定位后请求商品列表，检查返回结果中是否包含 `distance` 字段，并观察前端是否正确展示距离信息。随后调整检索半径，比较返回结果数量和商品排序是否符合预期。

在校园范围内，商品之间的距离通常较短，距离展示不需要达到测绘级精度。测试重点并不是验证每一个距离值与真实路径距离完全一致，而是验证球面距离计算逻辑稳定、半径过滤有效、距离排序方向正确。由于 Haversine 公式计算的是两点之间的球面直线距离，而用户实际步行路线可能受到道路和建筑影响，因此论文中将其定位为“附近检索和距离参考”，而不是精确导航。

交易闭环测试重点关注聊天、订单和评价之间的衔接关系。用户从商品详情页联系卖家后，系统应创建或复用会话，并生成与该商品相关的意向订单。消息发送后，会话列表应展示最后一条消息和未读状态。订单确认完成后，商品状态应更新为已售，同一商品的其他进行中订单应被取消。评价提交后，评价记录应写入数据库，被评价用户信用分应根据星级发生变化。通过这一组测试，可以验证系统已经从“商品展示平台”扩展为“可记录交易过程的平台”。

从测试结果看，系统能够完成校园二手交易的主要业务流程。用户端可以完成登录、发布、浏览、附近检索、详情查看、收藏、留言、聊天、订单和评价操作；后台端可以完成登录、统计查看、用户管理、商品审核、举报处理和日志查看。LBS 附近检索能够根据用户坐标返回距离字段，并在列表中展示距离信息；交易闭环能够从聊天进入订单，并在确认完成后提交评价和更新信用分。

系统在演示环境下的主要风险集中在定位授权、局域网访问和 WebSocket 连接三个方面。若用户拒绝定位，系统会回退到普通商品列表；若小程序端无法访问 `localhost`，需要使用本机局域网 IP；若 WebSocket 连接异常，系统可通过 HTTP 接口加载历史消息和发送消息。上述处理能够提高毕业设计演示过程的稳定性。

从测试充分性看，本文测试覆盖了毕业设计要求下的主要功能，但仍属于中小规模系统测试。由于没有真实上线用户和长期运行数据，测试结果主要反映本地演示环境下的功能正确性。若系统后续投入真实校园使用，还需要补充并发访问测试、长时间 WebSocket 稳定性测试、异常网络测试、数据库备份恢复测试和更多真实用户可用性测试。

### 5.7 系统不足分析

系统已经能够完成毕业设计要求下的主要流程，但仍存在一些不足。首先，附近检索目前主要在应用层完成距离计算，适合校园中小规模数据，若后续数据量明显增加，需要考虑空间索引或数据库地理计算能力。其次，商品审核在演示版本中默认通过，虽然后台具备审核能力，但正式上线时应改为发布后待审核。再次，聊天模块目前以文本消息为主，图片消息、已读回执和断线重连仍可继续增强。最后，信用分规则较为简单，尚未引入更多行为数据和复杂风控策略。

## 第6章 总结与展望

### 6.1 工作总结

本文设计并实现了一套基于 LBS 的校园二手交易微信小程序。系统使用 Taro、React 和 TypeScript 构建用户端，使用 Flask、SQLAlchemy 和 MySQL 构建后端服务，使用 Vue 3 构建后台管理端，并通过 JWT 和 WebSocket 支撑身份认证与即时通信。围绕校园线下面交场景，系统实现了商品发布、附近检索、地图展示、商品详情、收藏、浏览历史、留言、聊天、订单、评价、信用分、举报和后台治理等功能。

本文的核心工作在于将位置能力和交易闭环结合起来。通过 Haversine 公式计算用户与商品之间的距离，系统能够按照校园附近范围展示商品；通过聊天、订单、确认完成和评价信用，系统能够记录交易过程并形成反馈；通过后台管理端，系统能够对用户、商品、举报和评价进行管理，增强了平台完整性。

### 6.2 项目特点

本文项目具有以下特点：

1. 面向校园近场交易场景，突出 LBS 附近检索、距离排序和地图展示。
2. 采用微信小程序作为用户入口，符合校园用户轻量化使用习惯。
3. 将商品详情、聊天、订单、完成确认、评价和信用分串联为轻量交易闭环。
4. 实现独立后台管理端，使系统具备商品审核、举报处理和操作日志能力。
5. 引入 AI 发布辅助和发布前预检，提高商品发布信息的完整度。

### 6.3 后续展望

后续系统可从以下方向继续优化。第一，引入空间索引或数据库地理计算能力，提高大规模商品数据下的附近检索效率。第二，完善商品审核流程，使用户发布、后台审核、结果反馈形成更严格的内容治理闭环。第三，增强聊天能力，支持图片消息、消息已读状态和断线重连。第四，优化信用分模型，将交易完成率、举报结果、评价内容和用户行为纳入综合计算。第五，将 AI 辅助能力扩展到图片识别、违规内容检测和价格区间分析，使商品发布质量进一步提升。

综上所述，本文完成了基于 LBS 的校园二手交易微信小程序的设计与实现，并对系统的核心功能、位置检索能力、交易闭环和后台治理能力进行了验证。该系统能够为校园闲置物品流通提供轻量化平台，也为类似校园服务系统的开发提供参考。
