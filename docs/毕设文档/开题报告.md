## 一、研究背景

随着移动互联网的发展、共享经济理念的普及以及高校学生消费方式的变化，校园闲置物品的流转需求持续增加。教材、数码产品、生活用品、体育器材等物品在学生群体中具有较高的更新频率，而传统的线下交易方式大多依赖熟人介绍、班级群转发、朋友圈发布或校内临时摆摊。这类方式虽然进入门槛低，但也存在信息分散、检索效率低、传播范围有限、成交链路不完整等问题。对于买方而言，往往需要在多个社交渠道中反复筛选信息；对于卖方而言，商品曝光和精准匹配效率也比较有限。因此，从校园真实使用场景出发，设计一个兼顾发布、检索、沟通、交易与评价的二手交易平台，具有较强的现实需求。

从技术发展趋势看，位置服务（Location-Based Services, LBS）已经成为移动应用的重要能力之一。地理信息系统与位置服务相关研究，已经从早期的位置表达、地图显示和空间数据组织，逐步扩展到空间分析、位置感知、移动决策支持和行业应用等多个方向。[1][2][3] 龚健雅系统阐述了地理信息系统的基本概念、空间数据组织方式与空间分析基础，为位置相关系统设计提供了理论支撑。[1] 李建松从地理信息系统原理、空间数据处理与分析方法出发，对 GIS 在信息系统中的应用逻辑进行了梳理。[2] 马驰则进一步结合实际应用场景，说明了 GIS 与位置服务在工程设计和行业系统中的结合方式。[3] 这些研究为本课题将地理位置能力引入校园交易场景提供了较为明确的理论基础。

对于校园二手交易这一具体场景而言，LBS 的价值主要体现在“缩短交易半径”和“提高匹配效率”两个方面。传统综合型二手平台虽然功能较为丰富，但往往面向全国或全城用户，交易链条更长、物流和面交成本更高，未必适合校园内部的近距离流转需求。校园交易具有明显的本地化特征，用户更在意商品是否离自己近、是否便于线下见面、是否来自同校用户，以及交易过程是否足够简洁安全。基于此，如果平台能够结合用户当前位置、商品发布位置和距离排序逻辑，将“附近商品”“就近沟通”“本校交易”等能力整合起来，就能比单纯按照发布时间展示商品的模式更贴近校园场景的实际需求。

从国内有关校园二手交易与二手平台的研究来看，近几年学者更关注大学生群体中的闲置流转需求、平台建设方式以及交易制度问题。闫锦彪、杨冬梅、张进围绕校园二手教材交易场景，提出通过平台化建设提升教材流通效率，并从系统实现角度分析了二手教材网络交易平台的建设方法。[8] 贺欢、胡成耀、汪洋、谢腾飞通过对大学生二手交易平台的调查分析，指出大学生已经成为二手交易市场中的活跃群体，平台发展前景与交易制度、用户需求和运行机制密切相关。[11] 孙袁缘则从平台治理角度出发，分析了二手交易平台内部纠纷处理和规则治理的现实问题，说明二手交易平台除了信息发布和交易撮合之外，还需要考虑平台信用、规则约束与纠纷处理能力。[10] 这些研究说明，二手交易平台并不是简单的信息发布系统，而是一个需要同时处理匹配效率、信任机制、用户体验和平台治理的问题域。

国内关于校园二手交易与小程序应用的研究虽然已有一定积累，但大多集中在单一模块设计或通用电商框架迁移上。除校园二手教材交易平台研究外，[8] 陈瑞斯围绕微信小程序在学习平台中的应用展开研究，指出小程序具备无需下载安装、依托微信生态、便于推广和适合轻量业务场景的特点。[9] 马欣竹、王力华在对小程序行业应用进行综述时，也强调了小程序在低门槛接入、场景适配和服务触达方面的优势。[7] 从应用层面看，小程序已经成为校园服务场景中较容易落地的开发形态，这也使得基于微信小程序构建校园二手交易系统具有较强的可行性。

相比传统 APP 或网站，小程序在校园场景中有几个现实优势。第一，用户进入成本较低，不需要额外下载安装，更符合学生“即用即走”的使用习惯。第二，小程序可以较方便地调用定位、地图、网络请求和本地缓存等能力，适合和 LBS 功能结合。第三，小程序依托微信生态，天然适合社交传播和熟人扩散，在校园内部更容易形成用户基础。[4][5][6][7][9] 因此，从平台入口来看，采用微信小程序作为前端载体是比较合理的技术选择。

本课题拟解决的核心问题主要包括以下几个方面。其一，如何针对校园二手交易场景构建完整的业务闭环，而不是停留在商品展示层面。系统不仅需要支持商品发布和浏览，还应覆盖搜索筛选、附近检索、收藏、聊天、订单、评价、举报等功能。其二，如何将 LBS 能力有效融入二手交易流程。这里不仅仅是“显示位置”，更包括根据经纬度进行距离计算、范围筛选、距离排序和地图展示，使平台真正具备本地化检索能力。其三，如何在用户端之外兼顾平台治理。校园交易虽然规模相对有限，但仍然会涉及虚假发布、违规内容、纠纷举报等问题，因此平台还需要预留商品审核、举报处理、用户封禁、分类维护和日志记录等后台管理能力。

从技术要求看，本课题应达到以下目标：一是构建一个能够在微信小程序环境下运行的前端系统，完成用户登录、商品管理、交易沟通和个人中心等交互功能；二是设计并实现稳定的后端服务，支持用户、商品、订单、消息、评价、收藏、举报等核心业务；三是实现基于经纬度的附近检索功能，并采用距离计算方法完成商品与用户位置之间的空间排序；四是完成数据库模型设计，保证系统核心业务数据之间具有清晰的实体关系；五是补充后台管理系统的设计方案，使整个平台具备一定的审核与治理能力。

本课题的研究意义主要体现在理论和实践两个方面。从理论层面看，课题将 LBS、本地化交易、小程序应用和平台治理放在同一个系统中综合考虑，不是简单套用传统电商模板，而是围绕校园场景对位置服务与交易流程之间的关系进行工程化实现。这种研究方式虽然偏应用，但能够较完整地反映移动信息系统、位置服务与电子商务系统的交叉特点。[1][2][3] 从实践层面看，校园闲置物品流转是一个真实且长期存在的需求场景，系统如果实现得当，可以为学生提供更便捷的交易渠道，也有助于提升闲置资源再利用效率。[8][10][11]

结合当前项目的已有基础，本课题的主要工作内容包括：分析校园二手交易业务需求；设计系统总体架构与数据库结构；完成微信小程序前端页面开发；实现 Flask 后端接口与业务逻辑；构建基于 LBS 的附近检索与地图展示能力；设计聊天、订单、评价、举报等交易闭环模块；补充后台管理系统需求与实现方案；最后完成系统测试、部署与论文撰写。整体上，本课题属于一个面向真实场景的应用型系统设计与实现项目，既强调功能完整性，也强调一定的技术特色和可扩展性。

## 二、毕业论文（设计）研究方案

本课题采用“需求分析—总体设计—详细设计与实现—测试优化—总结撰写”的研究路线开展。

第一阶段为需求分析阶段。结合校园二手交易的实际场景，对用户发布商品、附近检索、沟通交易、收藏浏览、订单评价、举报反馈等需求进行梳理，并补充管理员在商品审核、举报处理、用户管理和分类维护方面的需求，明确系统边界、角色划分与主要业务流程。

第二阶段为系统总体设计阶段。根据前后端分离思想设计系统架构，前端采用微信小程序作为用户入口，后端采用 Flask 提供 RESTful API 服务与 WebSocket 即时通信扩展，数据库采用 MySQL 进行业务数据持久化。与此同时，完成用户、商品、分类、订单、会话、消息、评价、举报、收藏、浏览历史、管理员和操作日志等实体关系设计，形成数据库结构与接口设计方案。

第三阶段为系统实现阶段。前端基于 Taro + React + TypeScript 实现首页、发布页、详情页、个人中心、聊天页、订单页、收藏页、浏览历史页和评价页等页面；后端基于 Flask + SQLAlchemy 实现用户认证、商品管理、上传、消息、订单、评价、举报等接口；通过 JWT 完成登录鉴权；通过统一请求封装实现前后端交互。

第四阶段为关键技术实现阶段。针对本课题的特色需求，在商品列表和附近检索接口中引入基于 Haversine 公式的距离计算逻辑，支持根据用户当前位置对商品进行范围筛选、距离排序和地图展示。同时，结合交易流程实现会话建立、消息发送、订单创建、订单完成和双向评价功能，保证系统形成较完整的交易闭环。

第五阶段为平台治理设计阶段。基于后台管理系统需求文档，补充管理员登录、工作台总览、商品审核、举报处理、用户封禁、分类维护和操作日志等模块设计，使系统从单纯的用户端交易应用扩展为具备一定平台治理能力的完整系统。

第六阶段为测试与优化阶段。围绕用户登录、商品发布、附近检索、聊天、订单评价和举报处理等功能设计测试用例，验证系统在功能完整性、接口可用性和页面交互方面的表现；针对测试过程中发现的问题，对接口返回、异常处理和页面交互细节进行修正和优化。

本课题拟采用的研究方法主要包括：

1. 文献研究法。通过查阅位置服务、在线二手交易、小程序应用及本地化交易平台相关文献，为系统定位、功能设计和研究背景提供理论支持。
2. 需求分析法。结合校园场景特点，对目标用户、使用方式和交易流程进行分析，明确系统建设目标和功能范围。
3. 系统设计法。运用软件工程方法完成系统架构设计、数据库设计、接口设计和模块划分。
4. 实验与测试法。通过功能测试和场景化测试验证系统主要模块的正确性与可用性。

技术路线可概括为：微信小程序端负责页面展示与交互，后端负责业务逻辑与数据处理，数据库负责持久化存储，LBS 算法负责空间距离计算，后台管理端负责平台治理与审核。论文写作中将围绕这一技术路线展开需求、设计、实现和测试四个层面的分析。

## 三、毕业论文（设计）预期成果及创新点

### 3.1 预期成果

1. 完成一套可运行的基于 LBS 的微信小程序校园二手交易系统，具备用户登录、商品发布、附近检索、地图展示、聊天沟通、订单管理、交易评价、收藏、浏览历史和举报反馈等核心功能。
2. 完成系统后端服务与数据库设计，实现用户、商品、订单、消息、评价、举报等核心业务的接口支持。
3. 完成后台管理系统设计方案，具备商品审核、举报处理、用户管理、分类维护和日志管理等平台治理思路。
4. 完成毕业论文正文撰写，包括研究背景、系统分析、总体设计、详细实现、系统测试、总结与展望等内容。
5. 形成相应的过程性文档和交付材料，如需求文档、技术架构设计文档、数据库设计文档、部署文档和测试材料等。

### 3.2 创新点

1. 将 LBS 能力直接引入校园二手交易系统中，不再仅按发布时间展示商品，而是结合用户当前位置实现附近检索、距离排序和地图展示，突出“就近交易”的场景特点。
2. 将微信小程序与校园二手交易场景结合，利用小程序轻量化、低门槛和便于传播的特点，提升系统在校园环境中的适配性。
3. 在普通商品展示与浏览功能之外，补充聊天、订单、评价、举报等模块，使系统形成较完整的交易闭环，避免项目停留在简单的信息发布层面。
4. 在用户端系统基础上引入后台管理系统设计，补充商品审核、举报处理和平台治理能力，使项目整体结构更接近真实平台应用。
