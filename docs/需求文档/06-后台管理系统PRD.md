**项目名称**：基于 LBS 的校园二手交易平台后台管理系统  
**文档版本**：V1.1  
**编写日期**：2026-03-16  
**适用阶段**：毕业设计开发与答辩阶段  
**关联前台系统**：微信小程序端 `miniapp`、业务服务端 `backend`

---

## 1. 文档说明

### 1.1 编写目的

本文档用于明确校园二手交易平台后台管理系统的建设目标、业务范围、功能需求、权限设计、关键流程、非功能要求与验收标准，为后台管理端的设计、开发、测试、答辩演示提供依据。

### 1.2 背景说明

当前项目已完成微信小程序用户端的主要业务闭环，包括：

- 用户登录与资料维护
- 商品发布、编辑、上下架、浏览
- LBS 附近检索与地图展示
- 收藏、浏览历史
- 聊天会话与消息
- 订单创建与完成
- 评价与信用分
- 举报提交

但当前系统主要面向普通用户，缺乏平台管理能力。为了满足毕业设计对“系统完整性、平台治理、业务闭环、安全控制”的要求，需要补充后台管理系统，用于管理员对用户、商品、订单、举报、评价、分类、数据统计等进行统一管理。

### 1.3 产品定位

后台管理系统是平台运营和治理中心，面向管理员和审核人员提供：

- 平台数据总览
- 内容审核与违规处理
- 用户管理与风险控制
- 商品和订单运营管理
- 举报处理与评价管理
- 分类配置与基础设置
- 日志追踪与权限控制

### 1.4 技术实现建议

为控制开发成本并保证后台界面的成熟度，后台管理端建议采用以下技术方案：

- 前端框架：Vue 3
- 构建工具：Vite
- 开发语言：TypeScript
- 路由管理：Vue Router
- 状态管理：Pinia
- UI 组件库：Element Plus
- 图表组件：ECharts
- 网络请求：Axios
- 后端接口：复用现有 Flask 项目，新增 `/api/admin/*` 路由

技术选型原因如下：

1. Vue 3 生态成熟，适合快速搭建中后台页面。
2. Element Plus 与后台管理系统场景契合度高，表格、表单、弹窗、分页等组件完备。
3. Vite 启动和构建速度快，适合毕设项目快速开发与演示。
4. 可在不改动现有小程序前端的前提下，独立新增一个后台 Web 管理端。

## 2. 产品目标

### 2.1 业务目标

1. 建立平台治理能力，支持对用户、商品、举报、评价等内容进行审核和干预。
2. 完善系统业务闭环，使项目从“用户端应用”升级为“完整平台系统”。
3. 为毕业设计答辩提供更完整的产品形态与管理流程支撑。

### 2.2 产品目标

1. 提供清晰易用的后台工作台，支持管理员高效处理日常运营任务。
2. 支持按角色进行权限控制，避免普通管理员越权操作。
3. 支持数据统计展示，帮助管理员了解平台运行状态。

### 2.3 答辩目标

1. 体现系统不仅具备前台交易功能，也具备后台治理能力。
2. 能够展示“平台如何处理违规商品、恶意用户、举报内容、订单纠纷”等关键场景。
3. 增强系统完整性、可管理性和现实应用价值。

## 5. 核心问题与建设价值

### 5.1 当前存在的问题

1. 平台缺少后台治理入口，违规内容无法统一处理。
2. 举报提交后没有正式处理闭环。
3. 商品审核目前默认通过，不利于答辩展示“平台审核机制”。
4. 缺少运营视角的数据看板，不便展示项目整体运行效果。
5. 管理员行为缺少日志记录，不利于追踪责任。

### 5.2 后台建设价值

1. 增强平台安全性和可控性。
2. 提升系统完整度，更符合毕业设计项目要求。
3. 增加论文可写内容，包括角色权限、内容审核、平台治理、后台架构设计。

## 6. 业务流程概述

### 6.1 商品审核流程

1. 用户发布商品
2. 商品进入待审核状态
3. 管理员在后台查看待审核商品
4. 管理员选择通过或驳回
5. 若通过，商品在前台列表可见
6. 若驳回，记录驳回原因，前台不可展示

### 6.2 举报处理流程

1. 用户在前台提交举报
2. 举报进入后台待处理列表
3. 管理员查看举报对象、原因、补充说明
4. 管理员选择处理结果：驳回、确认违规、封禁用户、下架商品等
5. 系统记录处理人、处理时间、处理备注

### 6.3 用户风控流程

1. 管理员在用户列表查看账号信息、信用分、发布情况、举报情况
2. 发现异常用户后可执行封禁/解禁
3. 封禁后前台用户无法继续进行受限操作

### 6.4 订单运营流程

1. 管理员查看订单列表及状态
2. 处理异常订单或核查纠纷订单
3. 结合评价、举报信息辅助判断风险

## 8.2 管理员登录与认证

### 8.2.1 功能目标

保证后台仅允许授权管理员访问，支持基础账号密码登录与身份校验。

### 8.2.2 功能列表

| 编号 | 功能项 | 说明 | 优先级 |
|------|--------|------|--------|
| AU-001 | 登录 | 管理员通过账号密码登录后台 | P0 |
| AU-002 | 登录态管理 | 登录成功后签发 token，并维持会话状态 | P0 |
| AU-003 | 退出登录 | 主动退出后台 | P0 |
| AU-004 | 权限校验 | 不同角色访问不同菜单与接口 | P0 |
| AU-005 | 密码修改 | 管理员修改个人密码 | P1 |

### 8.2.3 规则说明

1. 管理员账号仅由超级管理员创建。
2. 后台接口必须进行管理员身份认证，不能复用用户端 token。
3. 登录失败需提示账号或密码错误。
4. 可记录最近登录时间。

| GM-002 | 商品详情 | 查看商品完整信息、图片、发布人、位置 | P0 |
| GM-003 | 商品审核 | 审核待上架商品，通过或驳回 | P0 |
| GM-004 | 强制下架 | 对违规商品进行下架处理 | P0 |
| GM-005 | 删除商品 | 逻辑删除或物理删除商品 | P1 |
| GM-006 | 商品状态筛选 | 待审核、已上架、已下架、已售出 | P0 |
| GM-007 | 驳回原因记录 | 商品驳回时填写原因 | P1 |

### 8.4.3 列表字段建议

- 商品ID
- 标题
- 分类
- 价格
- 发布用户
- 校内区域/位置
- 浏览量
- 商品状态
- 审核状态

### 8.4.4 业务规则

1. 商品审核状态建议定义为：待审核、已通过、已驳回。
2. 商品业务状态建议定义为：上架、下架、已售出。
3. 被举报较多的商品应支持快速定位与优先处理。
4. 管理员强制下架后，商品前台不可见。

### 8.4.5 答辩建议

若时间允许，建议将前台“发布即通过”改为“发布后待审核”，后台审核后才展示。这样后台价值会更强，也更符合平台治理逻辑。

## 14. 非功能需求

### 14.1 性能要求

1. 后台列表页查询响应时间应控制在 2 秒内。
2. 工作台总览接口响应时间应控制在 2 秒内。
3. 高频列表支持分页，默认每页 10 到 20 条。

### 14.2 安全要求

1. 后台接口必须要求管理员身份认证。
2. 密码不得明文存储。
3. 敏感操作应记录日志。
4. 不同角色必须进行权限隔离。

### 14.3 可用性要求

1. 页面布局清晰，适配常见桌面浏览器。
2. 操作后必须给出成功/失败反馈。
3. 删除、封禁、下架等操作需二次确认。

### 14.4 可维护性要求

1. 模块划分清晰，接口命名统一。
2. 管理后台与用户端后端接口保持隔离。
3. 后台日志与业务日志可追溯。

## 15. 风险与约束

### 15.1 项目风险

1. 若后台功能范围过大，可能影响毕设交付进度。
2. 若商品审核机制全面切换，可能联动修改前台商品流转逻辑。
3. 若权限系统设计过重，会增加开发与调试成本。

### 15.2 控制建议

1. 优先完成 P0 功能，先保证系统闭环。
2. 权限系统先做角色级，不做复杂按钮级权限。
3. 趋势图、通知、管理员管理可作为增强项逐步补充。

## 17. 验收标准

### 17.1 功能验收

1. 管理员可成功登录后台并访问有权限的菜单。
2. 后台可查看平台核心数据总览。
3. 管理员可查看用户列表并执行封禁/解禁。

5. 管理员可处理举报并记录处理结果。
6. 管理员可查看订单列表与详情。
7. 管理员可维护商品分类。
8. 系统能记录后台关键操作日志。

### 17.2 业务验收

1. 用户提交举报后，后台可见并可处理。
2. 商品进入待审核后，审核通过才可在前台展示。
3. 被封禁用户无法继续进行受限操作。
4. 管理员处理记录具备可追溯性。

### 17.3 答辩验收

1. 可现场演示管理员登录。
2. 可演示待审核商品处理。
3. 可演示举报处理与联动下架。
4. 可演示用户封禁。
5. 可演示工作台数据统计。

## 18. 毕设落地建议

为了兼顾开发成本与答辩效果，建议后台系统按以下策略落地：

1. 前端技术建议独立使用 Vue 3 + Vite + TypeScript + Element Plus，开发效率高，后台风格成熟，适合毕设演示。
2. 后端在现有 Flask 项目中新增 `/api/admin/*` 路由即可，不必重写服务。
3. 首先打通“管理员登录 -> 工作台 -> 商品审核 -> 举报处理 -> 用户封禁 -> 日志记录”主线。
4. 如果时间不多，趋势图、通知管理、评价管理都可以先做简版。

结论：

后台管理系统不宜追求复杂，而应突出“平台治理闭环”。只要把审核、举报、封禁、统计、日志这几项做好，已经足够支撑毕业设计项目的完整性和答辩展示效果。
